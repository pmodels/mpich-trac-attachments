Alexander,
 
MPI 2.1 (and 2.2) does not make any *changes* in the standard but both of them may and have resolved some ambiguities in the standard.  In some cases, existing tests (like the Intel tests) made certain assumptions about the behavior (particularly what constituted an error) that were not specified by the standard and that were eventually resolved in a way that was different from the test suite.  This reflects a bug in the test suite - the tests should have allowed any consistent reading of the standard.  The changes to the topology routines were in this group.  This was discussed as part of the voting process; in fact, the behavior of different MPI implementations at these corner cases was reported and the Forum knew that some implementations would need to change to conform to the more precise specification. 
 
For the others, for MPI_MAX_ERROR_STRING, I'm sure that the increase was because the previous length was too short for some error messages, though I would have recommended not changing it until MPICH2 1.1.  Rajeev and comment more on that change, I'm sure.  For the Fortran one-sided routines, the type of integer is specified in the standard, and if we had used MPI_Fint where the standard requires INTEGER (KIND=MPI_ADDRESS_KIND), we were fixing a bug in changing to MPI_Aint.  For the C++ interface, I don't know the specific answer - that may have simply been an oversight.  We could probably restore the previous order if that would help.  What was the exact change that caused problems?
 
Bill
